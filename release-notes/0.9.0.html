GeoTrellis 0.9 is now available!

A new major release of GeoTrellis, 0.9.0, is now available as a release candidate: 0.9.0-RC1. Please let us know if you find any issues with the release candidate so we can address them before the final release of 0.9.0.

New features of 0.9

Here's a sampling of the major changes that were made to the 0.9 release.

API refactor

There has been a major refactor to the API for GeoTrellis. Documentation on the new API is currently being written, so keep tuned for details. But in general, we're moving away from requiring users to manually create operations and pass in rasters as arguments, and instead having objects called 'DataSources' that represent the source of data, with the operations to transform or combine that data as methods off of those objects. These methods are not stateful, they return new DataSources per method call. Similar to the 'Future'-like model of Operations, transformations on a DataSource are not actually run until the server is told to 'run' the source, either explicitly or through a method call on DataSource and an implicit server parameter (see below for details).

Special thanks to joshmarcus for his vision of this API change and all the work he put into making it happen.

Here's some example code to show the change between 0.8 and 0.9:

0.8:

import geotrellis._
          import geotrellis.process._
          import geotrellis.raster.op.local

          val server = Server("example",Catalog.fromPath("/path/to/catalog"))

          val raster1 = io.LoadRaster("rater-layer-name")
          val multiply1 = local.Multiply(raster1, 5)

          val raster2 = io.LoadRaster("raster-layer-name-2")
          val multiply2 = local.Mulitply(raster2, 8)

          val added = local.Add(raster1,raster2)

          val minMax = added.map { r => r.findMinMax }

          server.get(added) match {
          case Complete((min,max),trace) => println(s"Min: $min, Max: $max")
          case Error(msg,trace) => println(s"Operation failed: $msg")
          }

0.9

import geotrellis._
          import geotrellis.process._
          import geotrellis.source._

          val raster = (RasterSource("raster-layer-name") * 5) +
          (RasterSource("raster-layer-name-2") * 8)

          val minMax = raster.minMax

          minMax.run match {
          case Complete((min,max),trace) => println(s"Min: $min, Max: $max")
          case Error(msg,trace) => println(s"Operation failed: $msg")
          }

This API change also means that upgrading any code that currently runs on an 0.8 release will probably be very broken. The ability to create and run Op[T]'s is still there, but some of the functionality, especially dealing with paralleling over tiles, was striped from them. Let us know on the mailing list or in #geotrellis on freenode IRC if you're upgrading and we'll lend a hand with the transition

Project restructure

One change that will definitely effect anyone using the geotrellis-server project is the change in the project structure. Where before the dependencies you would pull in was called geotrellis-server, that project is called geotrellis-jetty. That is because the server that was created with the WebRunner class was Jetty server; to be more agnostic to how users were creating their GoeTrellis endpoints, the more generic service code was pulled out into the geotrellis-services project, and the Jetty specific code was moved to geotrellis-jetty. An example of the flexibility that allows is that the admin project, which lets you view rasters out of the catalog through a local web application, was moved to use spray-routing to serve up requests.

IO, Warping

Reading ARGs from disk has been sped up, and in some cases, you'll find improvements of an order of magnitude or more. This is because during loading there was code to resample the raster along a cropped extent, so that rasters could be read off disk with a different RasterExtent then the raster was stored with. The thing was, this resampling code would run each time the raster was loaded off disk, even if you didn't want to resample; in this case the resampling code was called with the same raster extent in both the source and target positions. This amounted to a lot more work than had to be done: if you are reading the raster directly from disk, you can just read in the bytes, convert the byte array to an instance of RasterData, and be done. So loading a raster directly from disk becomes much faster, with the benefit increasing as the size of the raster loaded increases.

The functionality to resample the raster on load was very fast though, and so to have that code benefit not only rasters that were being loaded from disk or the cache, but also for rasters that were already loaded into memory, the Warp object was created. The name 'warp' is used to encapsulate a simultaneous crop and change of a resolution of raster. Now warping rasters from disk or in memory share code, and it benchmarks very fast. To warp a RasterSource, the call is now simply rasterSource.warp(newRasterExtent). To load a raster with a different raster extent, use RasterSource(LayerId("store-name","layer-name'), newRasterExtent).

Tile operation improvements

Running multiple operations over tiles has been greatly improved. For example, if you were to multiply a raster by an integer, add it to another tiled raster, and then run a zonal summary (such as taking the mean of values within a polygon) on that result, GeoTrellis 0.8 would unnecessarily reduce to a whole raster in memory between the different transformations (see issue #517). In 0.9, you'd get the desired behavior, where the multiplication, addition, and zonal summary all all done in parallel per tile, before the final zonal summary result is created from the reduction of the tile zonal summary results.

Clustering improvements

Along with improving how parallelizing operations over tiles works, there was also work to make it easier to distribute those operations over a cluster using Akka clustering. There's a .distribute call on DataSource which will distribute all of the operations of the DataSource's elements across the cluster, based on the configuration. For example,

val r =
          RasterSource("myraster")
          .localAdd(4) // add 4 to every cell
          .focalMin(Square(2)) // apply a focal min operation
          .tileHistograms // get a histogram for each of the tiles
          .distribute
          .converge

Will give a ValueSource[Histogram], which when run will give the histogram of the raster that was a result of the localAdd and focalMin calls. The .distribute call tells the GeoTrellis operation execution engine to distribute the operations across the cluster per tile. See joshmarcus's example in the ds-preview project for more information; this project and the documentation needs to be updated so be weary of things being a bit out of date before they are updated.

Macros!

The new geotrellis-macros project was created to deal with issue #624, based on the discussion of #324. This includes macros for checking whether a value is or isn't NoData, independent of what type that data is. And these checks are inlined through macro magic, so there's no performance hit for the nicer syntax. So

val r:Raster = getRaster(...) // Gets a raster somehow
          val i:Int = r.get(0,0) 
          val d:Double = r.getDouble(0,0)

          if(i == NODATA) { println("Is a NoData value." }
          if(i != NODATA) { println("Is a data value." }
          if(java.lang.Double.isNaN(d)) { println("Is a NoData value." } // or the slower, d.isNaN
          if(!java.lang.Double.isNaN(d)) { println("Is a data value.") }

should now be written as

val r:Raster = getRaster(...) // Gets a raster somehow
          val i:Int = r.get(0,0) 
          val d:Double = r.getDouble(0,0)

          if(isNoData(i)) { println("Is a NoData value." }
          if(isData(i)) { println("Is a data value." }
          if(isNoData(d)) { println("Is a NoData value." }
          if(isData(d)) { println("Is a data value.") }

Implicit server

A new GeoTrellis object (in the package geotrellis) was created that creates a geotrellis.process.Server instance to be used for any geotrellis operations. It's created with the catalog defined in your application.conf (the geotrellis.catalog setting). This means you don't have to spin up your own server. Also, importing geotrells._ will include an implicit server parameter set to that instance, so you can run a DataSource with dataSource.run instead of Geotrellis.server.run(dataSource), or if you want to get the value of the source, just call dataSource.get.
